# 安全策略

本仓库包含 AI/Agent 网关、防护规则、运行时回传、样本执行与报告导出能力。它会接触提示词、模型响应、访问令牌、审计日志和可能的敏感样本，因此安全问题请不要按普通缺陷公开处理。

## 支持范围

当前仅默认支持最新主线代码的安全修复与风险评估。以下内容不属于长期支持对象：

- 历史测试数据库
- 本地导出报告与备份快照
- 临时实验分支
- 已脱离主线的私有二次开发副本

## 报告方式

- 不要在公开 Issue 中直接贴出可复用漏洞利用链、真实密钥、真实业务数据或未脱敏报告。
- 优先通过你与维护者已有的私有沟通渠道提交。
- 如果计划把仓库公开发布到 GitHub，请在发布前启用 GitHub Private Vulnerability Reporting，或把本文件替换成组织实际安全邮箱。

建议报告内容：

- 影响模块
- 触发前置条件
- 最小复现步骤
- 影响范围与风险等级
- 是否涉及权限绕过、数据泄露、远程执行、审计缺失或防护失效
- 已做过的脱敏说明

## 敏感信息处理要求

提交漏洞材料前，请先移除或替换以下内容：

- `AI_API_KEY`
- `JWT_SECRET`
- `GATEWAY_API_TOKEN`
- 真实邮箱、手机号、身份证号、客户数据
- 完整的原始提示词或响应中包含的敏感片段

## 生产环境最低安全基线

如果你准备将本平台放入真实环境，至少先满足以下条件：

- 使用 PostgreSQL，而不是默认 SQLite
- 替换所有默认密码与默认令牌
- 设置强随机 `JWT_SECRET`
- 以 `APP_ENV=production` 运行
- 先执行：

```powershell
python .\backend\scripts\init_db.py --mode schema
```

- 再启动 API / Worker 运行进程
- 不要把 `backend/data/` 下的本地测试数据直接带入生产环境

## 已知安全边界

当前实现仍有明确边界，报告问题时请区分“设计边界”和“实现缺陷”：

- Worker 仍基于数据库轮询，不是独立消息队列
- 运行中任务的暂停 / 取消是协作式中断，不是强制上游硬中断
- 某些流式输出场景为避免脱敏前泄露，会切换为缓冲后输出
- PDF 报告等更正式的离线制品仍在持续补齐
