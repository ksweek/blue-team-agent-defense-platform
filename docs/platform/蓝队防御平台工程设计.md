# 蓝队防御平台工程设计

本文档聚焦当前代码仓库的真实工程结构、职责划分和演进方向。

截至 `2026-04-19`，项目已经具备完整的前后端原型、数据库落库、JWT/RBAC、统一日志、后台 Worker、OpenAI-compatible Provider 适配层，以及最小任务执行闭环。

## 1. 总体架构

当前工程采用前后端分离结构：

```text
Frontend (Vue 3 + TypeScript + Vite)
        |
        | HTTP / JSON + JWT
        v
Backend (FastAPI + Pydantic + SQLAlchemy)
        |
        | SQLite / PostgreSQL
        v
Data Models + Audit + Task Artifacts
        |
        | in-process worker
        v
OpenAI-compatible Provider
```

和早期“纯前端 + Mock”阶段相比，当前阶段已经有以下关键变化：

- 核心业务数据全部进入数据库模型
- 登录、JWT、RBAC 已接通
- 任务执行从同步模拟推进为后台 Worker 异步执行
- Provider 不再是本地假数据，支持真实模型调用
- 前端多个页面已统一成即时反馈式操作台

## 4. 当前交互模型

当前前端交互不是传统“编辑一堆字段 -> 点击保存”，而是统一为两类模式：

### 4.1 即时反馈动作

适用页面：

- 技能管理
- 资产保护
- 安全事件
- 系统设置动作区

特征：

- 点击动作立即提交到后端
- 顶部反馈条实时显示 `saving / saved / error`
- 结果会同步回写列表或焦点卡片

### 4.2 元数据驱动表单

适用页面：

- 防御配置
- 资产白名单
- 技能信任状态
- 系统设置

特征：

- 后端下发 `field_meta`
- 前端根据 `control / options / tone / helper_text` 渲染控件
- 不再由前端写死 `log_level`、`notify_email`、`risk_level`、`trust_status` 等映射

## 5. 任务执行链路

当前最小闭环如下：

```text
创建任务
  -> 任务入库，状态 queued
  -> /attack-tasks/{id}/run 提交
  -> task_worker 后台线程取任务
  -> task_runner 组装 prompt
  -> model_provider 调用 OpenAI-compatible 服务
  -> 保存 raw_response
  -> 生成 SecurityEvent
  -> 生成 Report
  -> 任务置为 done / failed
```

### 当前关键实现点

- `task_worker.py`
  - 负责恢复 `queued / running` 任务
  - 负责后台轮询与执行
- `task_runner.py`
  - 负责 prompt 构造、结果解析、事件和报告生成
- `model_provider.py`
  - 负责真实 Provider 调用
  - 记录调用开始、成功、失败日志

### 当前边界

- Worker 是进程内线程，不是独立任务系统
- 仅支持单一 Provider 协议类型
- 失败重试、取消、死信回收尚未实现

## 6. 日志与可观测性

当前本地开发模式已做统一日志收口：

- 应用生命周期：`app.lifecycle`
- 请求摘要：`app.http`
- Worker：`app.worker`
- 任务流水线：`app.pipeline`
- Provider：`app.provider`

后端本地启动入口：

```powershell
cd backend
python run_dev.py
```

这样 `uvicorn` 自身日志也能共用统一格式，不会和应用日志风格割裂。

## 8. 当前工程边界

### 8.1 仍偏原型的部分

- 报告导出已升级为 JSON / HTML / DOCX 真文件产物，但还没有 PDF 与签章版正式模板
- 系统动作已生成真实导出/备份文件，并提供系统工件列表与下载接口
- 没有专门的样本选择页或批量任务页

### 8.2 工程化尚未补齐的部分

- 没有 Alembic 迁移
- 没有 CI/CD
- 没有独立任务队列
- 已补最小回归测试，但还没有完整的 CI 自动化测试矩阵

### 8.3 安全能力尚未独立模块化的部分

- 当前没有独立 `security_engine/`
- 任务策略仍主要在 `task_runner.py` 内聚合
- 外部 Agent Runtime 事件回传接口还没形成完整标准

## 9. 推荐的下一步工程演进

1. 引入 Alembic，解决表结构演进问题。
2. 将 Worker 拆成独立进程，接入 Redis / RabbitMQ 一类队列。
3. 增加样本目录接口和批量任务执行能力，把数据集联调纳入平台主流程。
4. 把 Provider 抽象成可注册的多 Provider 适配层。
5. 将导出、备份、报告产物继续升级为更正式的 PDF / 签章式文件服务。

## 10. 一句话总结

当前项目最准确的工程描述是：

它已经完成了“前后端控制台 + 数据库主链路 + JWT/RBAC + 后台任务执行 + 统一日志 + 基本回归测试”的原型搭建，下一阶段的重点不再是继续堆页面，而是把任务系统、样本执行、运行时网关和生产级运维能力真正工程化。
