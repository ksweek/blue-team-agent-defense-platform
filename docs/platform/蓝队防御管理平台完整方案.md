# 蓝队防御管理平台完整方案

本文档从产品和实施视角描述当前项目的定位、已落地能力、适用场景和后续建设方向。

截至 `2026-04-19`，该项目最适合被定义为：

**一个面向 Agent / LLM 安全联调、策略治理和任务回归验证的蓝队防御管理平台原型。**

## 1. 建设目标

平台要解决的不是单点“拦截某条 Prompt”，而是把以下能力统一到一个控制台里：

- 防御策略配置
- 受保护路径 / 技能 / 插件治理
- 安全事件管理
- 攻击任务执行与追踪
- 报告归档
- AI 联调与攻击样本回归
- 审计留痕

## 2. 当前已经实现的能力

### 2.1 控制台与权限

- 登录页与 JWT 会话
- `admin / analyst` 双角色
- 页面级访问范围控制
- 接口级 RBAC

### 2.2 防御配置中心

- 防御项列表
- 单项/批量开关
- 模式切换：`off / observe / enforce`
- 防御策略扩展配置
- 受保护路径、技能、插件配置
- 后端字段元数据驱动表单

当前默认防御矩阵已经扩展为 8 条执行防线，覆盖项目内置攻击集的主要章节：

- 提示注入拦截
- 间接内容隔离
- 敏感工具最小授权
- MCP 会话与能力绑定
- 跨插件链路约束
- 长上下文污染防护
- 输出泄露脱敏
- 高风险动作审批校验

同时保留 guard / scan / advanced 三层细规则，用来补充：

- 风险来源追踪、记忆写入守卫、循环调用限幅
- 工具结果扫描、外部内容隔离扫描、敏感信息模式扫描
- 运行时联动审计与审批链路留痕

### 2.3 资产保护

- 资产列表
- 风险等级切换
- 保护状态切换
- 白名单规则新增 / 删除
- 白名单表单由后端元数据驱动

### 2.4 技能管理

- 技能列表
- 信任状态即时变更
- 扫描所选技能
- 扫描任务反馈回显

### 2.5 安全事件中心

- 事件列表
- 单条事件详情
- 单条状态处置
- 批量处置
- 命中规则、原始输入、处理日志回显

### 2.6 系统设置与动作执行

- 设置自动保存
- 后端字段元数据驱动
- 平台动作定义接口
- 动作颜色语义由后端下发
- 审计回显区

### 2.7 攻击任务闭环

平台已经具备最小可用闭环：

1. 创建任务
2. 后台 Worker 取任务
3. 调模型 / Agent Provider
4. 记录原始响应
5. 生成安全事件
6. 归档报告

## 3. 当前最适合的使用场景

### 3.1 本地联调

适合：

- 校验控制台是否可用
- 联调策略接口
- 联调前端自动保存和即时反馈
- 联调模型调用与任务闭环

### 3.2 攻击样本回归

适合：

- 跑 GitHub 收集的 Prompt Injection / Jailbreak / MCP / Plugin Chain 样本
- 观察策略是否命中
- 记录任务结果与事件回显

### 3.3 方案演示

适合：

- 向团队展示蓝队控制台长什么样
- 展示“治理页 + 执行页 + 审计页”三段式能力
- 演示最小闭环已经可以跑通

## 4. 当前不适合的场景

当前还不适合直接宣称用于生产运行时防护，原因包括：

- Worker 仍是进程内线程
- 还没有独立任务队列
- 报告导出虽已支持 JSON / HTML / DOCX 真文件，但还缺 PDF 和正式版模板体系
- 系统动作虽已生成真实备份/导出文件，但还缺对象存储、保留周期和异地容灾能力
- 样本执行还没有完整的页面化批量调度
- Provider 只实现了 OpenAI-compatible Chat Completions 协议

## 5. 角色与职责

### 管理员 `admin`

负责：

- 管理防御策略
- 管理系统设置
- 执行平台动作
- 查看所有管理页

### 分析员 `analyst`

负责：

- 查看仪表盘
- 查看和处置安全事件
- 管理资产、白名单、技能
- 发起技能扫描和攻击任务

## 7. 当前产品形态

从页面形态看，平台已经形成三类页面：

### 7.1 策略编辑型页面

- 防御配置
- 系统设置

特点：

- 后端元数据驱动
- 自动保存
- 顶部即时反馈

### 7.2 动作执行型页面

- 技能管理
- 资产保护
- 安全事件
- 系统设置动作区

特点：

- 点一下即执行
- 动作状态实时回显
- 列表和焦点面板联动

### 7.3 结果聚合型页面

- 仪表盘
- 任务反馈区
- 审计回显区

特点：

- 聚合查看
- 任务状态追踪
- 风险和审计集中呈现

## 8. 推荐实施路径

### 阶段一：把当前原型跑稳

目标：

- 保持现有页面与接口稳定
- 固化 `.env`、数据库和启动方式
- 完成基本冒烟测试和日志排障链路

### 阶段二：补可用执行面

目标：

- 增加样本选择能力
- 增加批量任务执行和回归报告
- 接外部 Agent Runtime 回传

### 阶段三：补工程化能力

目标：

- 引入 Alembic
- 引入独立 Worker
- 引入任务队列
- 完成 CI/CD 和测试体系

### 阶段四：补产物和集成

目标：

- PDF / 正式报告模板
- 备份/导出工件生命周期与对象存储
- 多 Provider / 多 Agent 接入

## 9. 当前最值得强调的结论

这个项目已经不是“只有 UI 的概念稿”，它已经具备：

- 真登录
- 真权限
- 真落库
- 真任务状态流转
- 真模型调用入口
- 真事件和报告记录

但它也还没有到“生产级安全平台”的程度。  
下一步最有价值的工作，是把统一代理网关、外部 Agent 回传、独立 Worker 调度和生产级报告体系继续做实，而不是继续增加新的静态页面。
