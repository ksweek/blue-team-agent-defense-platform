# 统一代理入口实施方案

本文档说明如何把当前蓝队防御平台进一步演进为“统一代理入口”，让需要保护的在线 AI 流量默认先经过平台，再被转发到真实模型或 Agent Runtime。

本文只给方案，不涉及当前代码修改。

## 1. 目标

当前项目已经具备三类基础能力：

- 受保护 AI 目标管理
- 前置授权与策略判定
- 运行时回调与结果归档

它已经适合做“任务执行控制台”和“联调评测平台”，但还不是一层真正的在线防护网关。  
要把它升级成“默认经过平台”的防护入口，核心不是继续堆页面，而是补齐一层统一代理链路。

统一代理入口要解决的不是“如何新增一个模型配置”，而是以下问题：

- 业务侧如何不再直连上游模型，而统一接入平台
- 平台如何在转发前做一次可审计的授权判定
- 平台如何按目标组、默认路由和策略决定上游去向
- 平台如何在转发后做输出脱敏、审计和事件生成
- 平台如何同时兼容纯模型 API 和 Agent Runtime

## 2. 目标状态

目标状态如下：

```text
Client / App / Agent
        |
        | HTTPS
        v
Blue Team Gateway
  - Client Auth
  - Request Normalize
  - Route Resolve
  - Policy Enforce
  - Optional AI Review
  - Output Redaction
  - Audit / Event / Report
        |
        | Upstream HTTP / SDK
        v
OpenAI / Azure OpenAI / vLLM / one-api / Dify / Coze / MCP Runtime
```

在这个形态下：

- 上游模型不再是业务直接调用的对象
- 平台成为所有在线请求的统一入口
- 防护不再只发生在“离线任务执行”阶段，而是进入“在线请求路径”

## 3. 为什么要做统一代理入口

如果没有统一代理入口，当前平台只能做到以下两种模式：

- 平台代为执行任务时，能做完整策略判定和审计
- 外部 Agent 主动接运行时回调时，能做部分授权和结果回传

但如果业务代码继续直接调用上游模型，例如：

```text
前端 / 业务服务 -> OpenAI API
前端 / 业务服务 -> one-api
自研 Agent -> vLLM
```

那么平台并不在请求链路里，它无法保证：

- 一定看得到原始输入
- 一定能阻断高风险请求
- 一定能脱敏输出
- 一定能落审计

所以要让“保护 AI”变成真实线上能力，必须让请求先走平台。

## 4. 统一代理入口的设计原则

### 4.1 兼容优先

北向接口尽量兼容主流调用方式，优先支持 OpenAI 风格请求，这样接入成本最低。

### 4.2 默认拒绝绕过

平台必须被定义为唯一允许的模型出口。  
如果业务仍能绕过平台直接调用上游，那防护就是可选项，不是控制面。

### 4.3 先标准化，再转发

无论北向进来的是 OpenAI 请求、Agent 执行请求还是 MCP 能力请求，都先转换成统一内部语义，再走授权和防护链。

### 4.4 策略和执行分离

策略负责判定“是否允许、是否复核、是否脱敏”，代理层负责“如何转发、如何超时、如何回退”。

### 4.5 证据链完整

每一次请求都要留下：

- 原始请求摘要
- 路由结果
- 授权结果
- 命中规则
- 上游响应摘要
- 脱敏结果
- 最终返回动作

## 5. 建议的总体架构

建议把统一代理入口拆成五层。

### 5.1 北向接入层

负责接收来自业务系统、前端、Agent 或第三方平台的请求。

建议提供两类北向 API：

- 模型兼容入口
- Agent 运行时入口

建议接口：

- `/gateway/v1/chat/completions`
- `/gateway/v1/responses`
- `/gateway/v1/agents/run`
- `/gateway/v1/runtime/authorize`
- `/gateway/v1/runtime/complete`

### 5.2 请求标准化层

负责把不同来源的请求转成平台内部统一结构。

统一后的请求对象至少应包含：

- `request_id`
- `tenant_id`
- `client_id`
- `source_type`
- `target_selector`
- `messages`
- `tools`
- `requested_scopes`
- `session_context`
- `metadata`

### 5.3 策略执行层

负责调用已有控制面能力做判定。

这一层是统一代理入口的核心，应复用当前已有能力：

- 规则风险评估
- 前置授权链
- AI 二次复核
- 输出脱敏

### 5.4 上游路由层

负责选择最终要调用的上游目标。

这里建议支持：

- 指定 `ai_endpoint_id`
- 指定 `endpoint_key`
- 按 `endpoint_group` 路由
- 未指定目标时走默认路由
- 失败时按策略是否回退到备用目标

### 5.5 审计与产物层

负责记录请求全过程并生成：

- 安全事件
- 任务记录
- 审计日志
- 报告产物
- 统计指标

## 6. 北向接口设计

### 6.1 模型兼容入口

最优先落地的是一个兼容 OpenAI 风格的代理入口。

建议最小接口集：

- `POST /gateway/v1/chat/completions`
- `POST /gateway/v1/responses`

请求体建议保留与 OpenAI 接口尽量一致的字段，例如：

- `model`
- `messages`
- `temperature`
- `max_tokens`
- `stream`
- `tools`
- `tool_choice`
- `metadata`

同时新增平台字段，建议通过 Header 或扩展字段传递：

- `X-Client-ID`
- `X-Route-Group`
- `X-Target-Endpoint`
- `X-Session-ID`
- `X-Approval-ID`
- `X-Trace-ID`

推荐策略是：  
尽量不要污染标准 OpenAI body，把平台特有字段放到 Header 中。

### 6.2 Agent 运行入口

这类入口适用于 Dify、Coze、自研 Agent、MCP Agent。

建议最小接口集：

- `POST /gateway/v1/agents/run`
- `POST /gateway/v1/runtime/authorize`
- `POST /gateway/v1/runtime/heartbeat`
- `POST /gateway/v1/runtime/complete`

`agents/run` 负责一次性代理整段 Agent 执行。  
`runtime/*` 负责细粒度控制，尤其适合工具调用和 MCP 能力调用。

## 7. 统一内部请求模型

建议平台内部引入一层标准请求对象，统一纯模型流量和 Agent 流量。

示意结构如下：

```json
{
  "request_id": "req_xxx",
  "client_id": "portal-prod",
  "tenant_id": "default",
  "source_type": "chat_completion",
  "target_selector": {
    "endpoint_id": 3,
    "endpoint_key": "corp-chat-prod",
    "endpoint_group": "生产"
  },
  "input": {
    "messages": [],
    "tools": [],
    "input_text": "",
    "requested_scopes": []
  },
  "runtime": {
    "session_id": "sess_xxx",
    "approval_id": "",
    "runtime_name": "self-agent",
    "source_plugin": "",
    "target_plugin": "",
    "mcp_server": "",
    "capability_name": ""
  },
  "metadata": {
    "user_id": "",
    "ip": "",
    "tags": []
  }
}
```

这层抽象的意义是：

- 北向协议可以不断扩展
- 策略层不用关心外部请求格式差异
- 上游层不用关心北向调用方是谁

## 8. 请求执行链

在线代理请求建议按以下顺序执行。

### 8.1 客户端鉴权

先认证调用平台代理入口的客户端，而不是直接信任任何能访问网关的人。

建议支持：

- 静态 API Key
- JWT
- 内网服务间签名

同时为每个客户端配置：

- 允许访问的目标组
- 默认目标
- 最大 QPS
- 最大并发
- 是否允许流式
- 是否允许工具调用

### 8.2 目标解析

根据请求头或 body 解析目标去向：

- 如果明确指定 `endpoint_key`，优先按它查
- 如果指定 `endpoint_group`，按组路由
- 如果都没指定，走当前默认路由

目标解析失败时应直接返回，不允许静默乱转发。

### 8.3 请求标准化

将原始请求转换为平台统一内部模型，并补齐：

- `request_id`
- `session_id`
- `trace_id`
- `client_id`
- `runtime metadata`

### 8.4 前置授权

调用现有策略控制面，对如下维度做判定：

- 提示注入风险
- 敏感目标路径
- 技能调用
- 插件调用
- MCP 能力绑定
- 高风险 scope 是否具备审批凭据

前置授权结果只允许三类：

- `allow`
- `review`
- `deny`

### 8.5 AI 二次复核

对规则未直接确认阻断，但存在风险的请求，可按策略进入 AI 复核。

建议沿用三档模式：

- `rules_only`
- `suspicious_review`
- `review_all_remaining`

### 8.6 上游转发

当请求被允许执行后，再转发到真实上游。

上游可能是：

- OpenAI
- Azure OpenAI
- vLLM
- one-api
- Dify Adapter
- Coze Adapter
- 自研 Agent Runtime

### 8.7 输出脱敏与回写

拿到上游结果后，不应直接原样返回。

必须先做：

- 敏感字段识别
- 凭据模式识别
- 输出脱敏
- 策略二次检查

必要时可将结果改写为：

- 局部脱敏后放行
- 全量拦截并给出安全提示
- 转人工复核

### 8.8 审计落库

最后写入：

- 请求摘要
- 授权结果
- 上游目标快照
- 输出摘要
- 命中规则
- 安全事件
- 报告产物

## 9. 南向上游适配策略

### 9.1 OpenAI

建议作为首个标准上游接入，优先支持非流式请求。  
原因是生态最通用、客户端最容易兼容。

### 9.2 vLLM

如果暴露 OpenAI 兼容接口，建议直接按 `openai_compatible` 处理。  
不要额外制造一套完全不同的北向协议。

### 9.3 one-api

建议把 `one-api` 当成“聚合上游”，不是安全控制面的替代品。

短期可以让平台代理到 `one-api`。  
长期建议把关键模型拆出来单独接，避免真实上游不可见。

### 9.4 Azure OpenAI

建议作为单独 provider 处理，而不是长期混用普通 `openai_compatible`。  
原因是路由、认证和 `api-version` 都和普通 OpenAI 不完全一致。

### 9.5 Dify / Coze

这类目标更适合通过 Adapter 模式接入。

推荐形态：

```text
Platform Gateway -> Dify Adapter / Coze Adapter -> Dify / Coze API
```

不要让平台直接背负过多第三方平台差异。  
适配复杂度高的部分，下沉到独立 Adapter 服务更稳妥。

### 9.6 自研 Agent / MCP Runtime

最适合通过统一运行时协议接入。  
能力调用粒度要尽量细，不要只在“整段会话结束后”才回传。

## 10. 必须补齐的数据模型

统一代理入口要真正落地，建议补齐以下实体或扩展字段。

### 10.1 GatewayClient

用于描述“谁有权访问代理入口”。

建议字段：

- `client_key`
- `client_secret_hash`
- `display_name`
- `status`
- `allowed_groups`
- `default_endpoint_id`
- `rate_limit_qps`
- `max_concurrency`
- `allow_stream`
- `allow_tools`

### 10.2 GatewayRequestLog

用于记录一条代理请求全过程。

建议字段：

- `request_id`
- `client_id`
- `source_type`
- `target_endpoint_id`
- `route_group`
- `authorization_decision`
- `ai_review_invoked`
- `upstream_status`
- `masked_output`
- `event_id`
- `created_at`

### 10.3 RoutePolicy

用于描述路由规则。

建议字段：

- `route_key`
- `match_type`
- `match_value`
- `primary_endpoint_id`
- `fallback_endpoint_ids`
- `timeout_seconds`
- `retry_policy`

### 10.4 ProxySession

用于跟踪多轮上下文和运行时会话。

建议字段：

- `session_id`
- `client_id`
- `target_endpoint_id`
- `risk_state`
- `last_authorization_decision`
- `last_event_id`

## 11. 流式输出的处理建议

流式输出是统一代理入口里最容易低估的部分。  
建议不要一开始就支持全量流式。

推荐分三阶段处理：

### 第一阶段

只支持非流式请求。  
先把授权、路由、审计、脱敏链跑通。

### 第二阶段

支持“缓冲式流式”。  
平台先聚合若干 token 或按句子切片，再做脱敏和转发。

### 第三阶段

支持真正增量流式审计。  
这时候要解决：

- 增量脱敏
- 中途截断
- 已发出内容不可回收
- 工具调用中间态回显

因此流式必须后置，不要作为第一阶段目标。

## 12. 网关部署建议

### 12.1 网络位置

建议把平台部署在所有上游模型出口之前：

```text
Client / App -> Blue Team Gateway -> Internet / Private AI Cluster
```

如果还有 `one-api` 这类聚合层，建议变成：

```text
Client / App -> Blue Team Gateway -> one-api -> Upstream Models
```

### 12.2 网络控制

为了避免绕过，建议配合：

- 出口白名单
- 安全组限制
- DNS/代理统一出口
- 只允许平台代理访问上游模型

如果业务服务仍可直接连上游，统一代理入口就只是建议路径，不是强制路径。

### 12.3 高可用

统一代理入口如果承担生产流量，必须考虑：

- 多实例部署
- 健康检查
- 限流
- 熔断
- 重试
- 超时控制
- 灰度发布

## 13. 实施阶段建议

### 阶段一：最小可用网关

目标：

- 只做 `POST /gateway/v1/chat/completions`
- 只支持非流式
- 只接 `openai_compatible`
- 只做单目标路由
- 跑通“鉴权 -> 授权 -> 转发 -> 落库”

验收标准：

- 业务已经可以通过平台代理成功调用模型
- 所有调用都有 `request_id`
- 所有调用都有授权结果和审计记录

### 阶段二：策略强化

目标：

- 接入 AI 二次复核
- 接入输出脱敏
- 接入目标组路由和默认回退
- 接入高风险 scope 审批

验收标准：

- 明确攻击请求会被阻断
- 可疑请求可以进入 AI 复核
- 命中敏感输出时可被脱敏或截断

### 阶段三：多上游扩展

目标：

- 新增 Azure OpenAI
- 支持 vLLM、one-api
- 支持按目标组和可用性路由

验收标准：

- 不同上游可纳入同一北向接口
- 平台可感知真实目标去向

### 阶段四：Agent Runtime 接入

目标：

- 补 `agents/run`
- 补 Adapter 方案
- 接 Dify / Coze / 自研 Agent / MCP Runtime

验收标准：

- 除纯模型 API 外，Agent 链路也能纳入统一控制面

### 阶段五：流式与生产化

目标：

- 支持流式
- 支持独立 Worker / 队列
- 完善观测和告警

验收标准：

- 可承载在线生产请求
- 有稳定的限流、超时和异常恢复能力

## 14. 关键风险

### 14.1 只做代理，不做网络封堵

这是最常见失败点。  
如果业务还可以直连上游，那统一代理入口就会被绕过。

### 14.2 一开始就做全流式和全平台兼容

这会直接把范围做炸。  
正确做法是先从非流式 `OpenAI-compatible` 开始。

### 14.3 把 Dify / Coze 当成首个主接入对象

这类平台适配差异大，最适合作为后续 Adapter 目标，不适合作为第一阶段主战场。

### 14.4 把策略逻辑写死在网关分支里

网关应复用统一控制面，不能把策略散落到多处。  
否则后面会出现“任务执行一套策略、在线代理又一套策略”的分裂。

## 15. 对当前项目的建议结论

结合当前仓库现状，最合理的演进顺序是：

1. 先把统一代理入口定义成一个新能力层，不替换现有任务链。
2. 第一阶段只支持 `OpenAI-compatible` 非流式请求。
3. 第二阶段复用现有策略链和 AI 复核链，不另起一套规则系统。
4. 第三阶段再补 Azure、vLLM、one-api。
5. 第四阶段把 Dify、Coze、自研 Agent、MCP 接到统一运行时协议上。
6. 最后再处理流式和生产级高可用。

一句话总结：

统一代理入口的本质，不是“再加一个 API”，而是把平台从“离线联调与任务执行控制台”，升级成“在线 AI 流量强制入口”。  
它的落地关键不在页面，而在三件事：

- 北向统一协议
- 中间统一策略链
- 南向统一上游路由与审计
